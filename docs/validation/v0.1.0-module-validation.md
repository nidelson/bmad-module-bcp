# Validação do Módulo — bmad-module-bcp v0.1.0 (VM)

> Relatório canônico em PT-BR ([ADR 0001](../ADR/0001-localization-strategy.md)).
> Fecha a Fase 4 do roadmap (slice 4/4: VM — Validate Module).

## Objetivo

Provar que o módulo **instala de verdade** num projeto BMAD limpo via o
installer oficial — a única garantia que o harness determinístico (suite
NFR, PR #13) não consegue dar, porque ele exercita os scripts diretamente,
nunca o caminho `npx bmad-method install`.

## Metodologia

Install não-interativo, source local (a árvore deste repositório), num
diretório descartável:

```bash
npx bmad-method@latest install \
  --directory <tmp> \
  --custom-source <repo-root> \
  --modules bcp \
  --tools claude-code \
  --yes
```

Formalizado como teste de integração guardado:
`tests/integration/test_install_e2e.py` (marker `integration`, roda só com
`BCP_E2E_INSTALL=1` — precisa de rede + npx, pesado para a suite unitária).

## Achado de conformidade (corrigido nesta slice)

A primeira execução do VM emitiu um aviso do installer:

> `bcp/module-help.csv header does not match canonical schema.`
> Esperado: `…,phase,preceded-by,followed-by,required,…`
> Encontrado: `…,phase,after,before,required,…` — *Data loaded positionally.*

O `module-help.csv` (e a constante `HEADER` em `merge-help-csv.py`) usavam
os nomes de coluna `after`/`before`. O schema canônico do BMAD 6.7.x espera
`preceded-by`/`followed-by`. Carregava posicionalmente (funcional), mas com
warning a cada install — ruído de conformidade num módulo distribuível.

**Causa raiz:** herdado do módulo irmão PULSE, que tem a mesma divergência.
Decisão: **não herdar o débito** — alinhar ao schema canônico. PULSE pode
reconciliar separadamente.

**Correção (nesta slice):**
- `skills/bmad-bcp-setup/assets/module-help.csv` — header renomeado.
- `skills/bmad-bcp-setup/scripts/merge-help-csv.py` — `HEADER` renomeado.
- Dados das linhas inalterados; colunas estavam vazias na prática.
- Suite NFR (128 testes) segue verde — leitura é por nome (`skill`,
  `menu-code`), não pelas colunas renomeadas.

Re-execução do VM pós-correção: **sem warning**, install limpo.

### Ausências esperadas pós-install (não são bugs)

- **`bcp-baseline.yaml` não semeado** e **hook `bmad-create-story.toml`
  não injetado** no install. Correto por design: `seed_baseline.py` e
  `inject_customize.py` rodam via a skill `/bmad-bcp-setup` (ação do
  usuário), não no install. O installer só cria diretórios e mescla config.
